# Multi-Tenant RAG Backend (Node.js + TypeScript)

Backend API for a multi-tenant RAG system where each tenant can upload and query only its own knowledge, with guardrails and vector search.

## Stack

- Node.js + TypeScript
- Express
- PostgreSQL + `pgvector`
- Google Gemini API (Embeddings + LLM)
- Vitest

Note: Assignment mentions Fastify; this implementation uses Express with equivalent API behavior and isolation guarantees.

## Setup

1. Install dependencies:

```bash
npm install
```

2. Configure environment in `.env`:

```env
PORT=4000
DATABASE_URL=postgres://<user>:<password>@<host>:<port>/<db>?sslmode=require
JWT_SECRET=<long_random_secret>
GEMINI_API_KEY=<your_gemini_api_key>
EMBEDDING_MODEL=gemini-embedding-001
CHAT_MODEL=gemini-2.5-flash
```

Note: keep `.env` secret values private.

3. Start the API:

```bash
npm run dev
```

Server: `http://localhost:4000`

## Database initialization

On startup, `initDb()` ensures:

- `vector` extension exists
- core tables are created
- vector indexes are created
- `chunks.embedding` is in `VECTOR` format for real model embeddings

## API Endpoints

Base URL: `http://localhost:4000`

1. `GET /health`
2. `POST /auth/signup` (creates tenant + admin user)
3. `POST /auth/login` (email/password)
4. `GET /auth/me` (requires bearer token)
5. `GET /tenant/:tenantId/documents`
6. `POST /tenant/:tenantId/documents` (admin/editor)
7. `DELETE /tenant/:tenantId/documents/:documentId` (admin)
8. `POST /tenant/:tenantId/query`
9. `GET /tenant/:tenantId/users` (admin)
10. `POST /tenant/:tenantId/users` (admin)

### Example request bodies

`POST /auth/signup`

```json
{
  "tenantName": "Infoware",
  "name": "Admin User",
  "email": "admin@infoware.com",
  "password": "StrongPassword123"
}
```

`POST /tenant/:tenantId/documents`

```json
{
  "fileName": "policy.txt",
  "content": "Refunds are allowed within 30 days with receipt."
}
```

`POST /tenant/:tenantId/documents` (PDF upload)

- Content-Type: `multipart/form-data`
- Form fields:
  - `file`: select a `.pdf` file
  - `fileName` (optional): custom document name

`POST /tenant/:tenantId/query`

```json
{
  "query": "What is the refund policy?"
}
```

## Guardrails

- Prompt injection keyword detection
- Out-of-scope query detection
- Low-confidence fallback response

## Tenant Isolation

- Every tenant-scoped query includes `tenant_id` filters.
- Retrieval joins enforce tenant consistency across tables.
- Cross-tenant document/query access is blocked by design.

## React Frontend

An interactive React frontend (with role-based UI access and animations) is available in `frontend/`.

1. Install frontend dependencies:

```bash
npm --prefix frontend install
```

2. Add env:

```bash
cp frontend/.env.example frontend/.env
```

3. Run backend and frontend in separate terminals:

```bash
npm run dev
npm run frontend:dev
```

Frontend URL: `http://localhost:5173`

Role behavior in frontend:
- Landing page with `Login` and `Signup`
- Signup creates a new tenant and first admin
- Login requires only `email` and `password`
- Only tenant `admin` can add users for that tenant
- `editor` and `viewer` are restricted by backend RBAC and tenant membership

## Render Deployment

This repo includes a Render Blueprint config: [render.yaml](d:\Multi Tenant RAG\render.yaml)

It provisions:
1. `multi-tenant-rag-api` (Node web service)
2. `multi-tenant-rag-frontend` (Static site)

### Steps

1. Push this repo to GitHub.
2. In Render, create a new Blueprint from that repo.
3. During setup, set real values for:
- `DATABASE_URL`
- `GEMINI_API_KEY`
4. After first deploy, update these env vars to your actual service URLs:
- Backend `FRONTEND_URL` = your frontend Render domain
- Frontend `VITE_API_BASE_URL` = your backend Render domain
5. Redeploy both services.

### Notes

- `JWT_SECRET` is auto-generated by Render from `render.yaml`.
- `PORT` is provided and used by backend.
- Frontend uses SPA rewrite to `index.html` for React Router routes.
